SOLID

- Single responsibility principle

- The Open Closed Principle

Class or function should be open for using and extending but not for modifying.

React lifecycles appropriate to Open-Closed Principle the same as decorators. (example_1)

Transaction classes appropriate to Open-Closed Principle. (example_2)

- The Liskov Substitution Principle

subtype that extends the basic class cannot be basic class of another subtype

Input Controller exists. It is basic type for searchPanl, DropdownWithSearch and Textarea. But SearchPanel cannot be basic for DropDown And Textarea (example_1)

AuthHandler exists. It is basic type for GoogleAuth, FacebookAuth and TwitterAuth. But GoogleAuth cannot be basic for FacebookAuth And TwitterAuth (auth_ex)

Request exists. It is basic type for User and Topics. But User cannot be basic for Topics (requests)

- The Interface Segregation Principle

If not every subtype can implemnents methods from abstract class, abstract class mustn`t consist it

xbox abstract class exist. It has getKinect method. But not every type of xbox has kinect, so we compelling to create stumbs.

authAbstract suppose to have some properties. But if we realize ownAuth method, the properties isn`t match with abstract method. So abstract method need to create right props in advance

- The Dependency Inversion Principle

High-level modelu should not depend on details of another module. And vice versa. They should depend ob abstraction

Send message class has message method inside. Message can be created with link. And when parse link, we should ensure that every type of link are accepted
